## 用户输入

```text
$ARGUMENTS
```

## 目标

将产品功能想法快速转化为简洁的功能概要文档，便于后续技术实现参考。

## 执行流程

### 1. 理解输入内容

用户可能提供：

- 功能页面描述（包含页面结构、字段、操作等）
- 简单的功能需求
- 业务场景说明
- 参考其他系统

### 2. 简单确认（仅在必要时）

**仅当用户输入不够清晰时**，才询问 1-2 个关键问题：

- 这个功能主要解决什么问题？
- 谁会使用这个功能？
- 有什么特殊的业务规则？

**如果用户输入已经很详细（如包含页面结构、字段、操作等），直接跳过确认，开始生成文档。**

### 3. 生成功能概要文档

#### 文档模板

```markdown
# [功能名称]

## 功能描述

一句话说明这个功能的目的和价值。

## 页面结构

### [页面名称]

**展示内容**：

- 说明页面展示什么数据

**列表字段**：

- 字段1：说明
- 字段2：说明

**操作按钮**：

- 操作1：点击后的行为
- 操作2：点击后的行为

**筛选/搜索**：

- 支持的筛选条件

### [页面名称2]（如有多个页面）

...

## 业务规则

列出关键的业务逻辑和限制条件：

- 规则1
- 规则2

## 数据关联

说明涉及的主要数据表和字段：

- 表名：主要字段
```

#### 文档保存位置

- 路径：`docs/产品文档/[功能名称]/[功能名称].md`
- 示例：`docs/产品文档/校友审核/校友审核.md`

每个功能一个独立目录，目录内放置同名的功能概要文档。

### 4. 更新文档索引

在 `docs/产品文档/README.md` 中添加新功能的链接：

```markdown
- [功能名称](./功能名称/功能名称.md)
```

保持简洁，不需要版本号、状态等额外信息。

### 5. 输出生成报告

简洁地告知用户：

```markdown
✅ 功能概要已生成

**文档位置**：`docs/产品文档/[功能名称]/[功能名称].md`

**包含内容**：

- 功能描述
- 页面结构
- 操作说明
- 业务规则
```

## 文档风格

### 简洁优先

- ❌ 不要：冗长的背景介绍、详细的用户旅程、复杂的验收标准
- ✅ 要：直接说明页面结构、字段、操作、规则

### 实用导向

- 重点是让开发人员快速理解要做什么
- 不需要 Given-When-Then 格式
- 不需要用户故事格式
- 直接列出页面、字段、操作即可

### 业务语言

- 用业务术语描述，不用技术术语
- 说清楚业务规则和限制条件
- 必要时说明数据来源和关联

## 使用示例

### 示例 1：详细输入（直接生成）

```bash
/spec.product-doc
校友审核列表
- 页面：表格页面，默认展示 status=Pending 的校友
- 字段：姓名、校友分类、毕业年份、提交时间
- 操作：查看详情、通过、驳回
- 筛选：按姓名、校友分类搜索
```

**AI 行为**：

- 不询问任何问题（输入已足够清晰）
- 直接生成功能概要文档
- 更新文档索引

### 示例 2：简单输入（需要确认）

```bash
/spec.product-doc 需要一个活动报名功能
```

**AI 行为**：

- 询问 1-2 个关键问题：
  - 活动报名的主要流程是什么？（用户看活动 → 报名 → 审核/确认？）
  - 有什么特殊限制？（人数限制、时间限制、审核要求？）
- 基于回答生成功能概要

## 注意事项

1. **保持简单**：不要过度设计，重点是快速记录功能要点
2. **避免冗余**：不需要 PRD、用户故事等复杂文档结构
3. **便于查阅**：按功能划分目录，方便后续查找和更新
4. **及时更新**：功能变更时，直接更新对应的功能概要文档
